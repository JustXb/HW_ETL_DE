# Стриминговый сервис — ETL-пайплайн и аналитические витрины

Этот проект моделирует простой аналитический пайплайн для стримингового сервиса.  
Основная задача — собрать данные из MongoDB, загрузить их в PostgreSQL и построить поверх них аналитические витрины.

В данных отражены основные активности пользователей платформы:

- пользовательские **сессии просмотра**
- **события воспроизведения видео**
- **обращения в поддержку**
- **рекомендации контента**
- **жалобы пользователей и модерация**



---

# Архитектура проекта

В проекте используются две базы данных.

## MongoDB (источник данных)

MongoDB используется как операционная база, где хранятся исходные события сервиса.

База данных streaming_final:

Коллекции:

- `WatchSessions` — сессии просмотра пользователей
- `PlaybackEvents` — события воспроизведения видео
- `SupportCases` — обращения пользователей в поддержку
- `ContentRecommendations` — рекомендации контента пользователям
- `AbuseReports` — жалобы пользователей и модерационные флаги

---

## PostgreSQL (аналитическое хранилище)

PostgreSQL используется как аналитическое хранилище данных.

База данных etl_final_warehouse:


Используются две схемы.

### Схема `stream_raw`

Это слой **сырых данных**, полученных из MongoDB.  
Данные здесь практически не агрегируются — они просто приводятся к табличному виду.

Таблицы:

- `watch_sessions_raw`
- `playback_events_raw`
- `support_cases_raw`
- `content_recommendations_raw`
- `abuse_reports_raw`

---

### Схема `stream_marts`

Это слой **аналитических витрин**.

Витрины содержат агрегированные показатели, которые можно использовать для аналитики .

Таблицы:

- `watch_activity_daily`
- `support_and_abuse_daily`

---

## DAG-и и задачи

### DAG `stream_ingest`

Этот DAG отвечает за загрузку данных из MongoDB в PostgreSQL. Данные загружаются в схему stream_raw.

`load_sessions` - Читает данные из WatchSessions. Во время загрузки выполняется обработка: вычисляется длительность сессии, определяется тип подписки пользователя (free, standard, premium), JSON-поля сериализуются. После обработки данные записываются в таблицу stream_raw.watch_sessions_raw

`load_events` - Загружает события воспроизведения из коллекции PlaybackEvents. Каждое событие содержит: тип события (play, pause, seek и др.), информацию о видео, позицию воспроизведения, качество видео. Данные сохраняются в таблицу stream_raw.playback_events_raw

`load_cases` - Загружает обращения пользователей в поддержку. Источник SupportCases. Дополнительно вычисляется: время решения тикета, количество сообщений в переписке. Результат записывается в stream_raw.support_cases_raw

`load_recommendations` - Загружает рекомендации контента для пользователей из ContentRecommendations. Рекомендации представляют собой список ID видео.

`load_reports` - Загружает жалобы пользователей из AbuseReports. Содержит:текст жалобы, рейтинг, статус модерации, флаги нарушений. Данные сохраняются в stream_raw.abuse_reports_raw

### DAG `stream_marts`
Этот DAG отвечает за построение аналитических витрин.

`mk_watch_activity` - Создаёт таблицу витрины `stream_marts.watch_activity_daily`. 

`fill_watch_activity` - По каждому пользователю и дате считает суммарное время просмотра (в минутах), число видео и долю премиум. 

`mk_support_abuse` - Создаёт таблицу витрины `stream_marts.support_and_abuse_daily`. 

`fill_support_abuse` - По дате и пользователю считает обращения в поддержку и жалобы, плюс время последнего обновления. 

# Итоговые витрины
## Витрина `watch_activity_daily`

Содержит агрегированную информацию о просмотре видео.

Метрики рассчитываются **по пользователю и дате**.

### Поля таблицы

- `activity_date`
- `user_id`
- `sessions_count`
- `total_watch_time_minutes`
- `videos_count`
- `premium_ratio`

### Что показывают метрики

- количество сессий просмотра у пользователя
- общее время просмотра видео
- сколько разных видео смотрел пользователь
- долю премиум-подписок среди сессий пользователя


---

## Витрина `support_and_abuse_daily`

Эта витрина объединяет данные:

- обращений пользователей в поддержку
- жалоб пользователей

Агрегация выполняется по: user_id + date


### Поля таблицы

- `snapshot_date`
- `user_id`
- `support_cases_count`
- `abuse_reports_count`
- `last_support_update`
- `last_abuse_report`

### Для чего используется витрина

Позволяет анализировать:

- активность пользователей
- пользователей с большим количеством жалоб
- нагрузку на службу поддержки
